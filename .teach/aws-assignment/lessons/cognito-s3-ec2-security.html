What is safe in S3 JavaScript, and what must stay on EC2

Your app already uses Cognito PKCE login and backend JWKS verification. This lesson walks through the security wiring for S3 frontend + EC2 backend, what the hardcoded values in cognito-auth.js actually mean, and what you still need to configure in AWS.

Public vs secret: the rule that stops panic

Seeing Cognito IDs hardcoded in frontend JavaScript feels wrong. For a browser app using a public client, most of those values are meant to be visible. The panic should be reserved for actual secrets.

Value Where it lives Safe in S3 JS? Why
clientId cognito-auth.js Yes Public SPA clients have no secret. AWS expects the client ID in browser requests.
userPoolId, region, hosted UI domain cognito-auth.js Yes These are identifiers, not credentials. Anyone can see them in network traffic.
Cognito client secret Never frontend Never Only for confidential/server clients. Your SPA uses PKCE instead.
DB_PASSWORD EC2 backend only Never Database access must never reach the browser.
JWT_SECRET EC2 backend (legacy fallback) Never Signing secret. Backend only, ideally in Secrets Manager.
Access / refresh tokens Browser storage after login Expected Stored in localStorage after login. Protect with HTTPS and XSS hygiene.
Report sentence: “The Cognito app client is configured as a public client with authorization-code grant and PKCE. Public identifiers such as the client ID and user pool ID are visible in frontend code by design; database passwords and JWT signing secrets remain on the backend.”
What S3 cannot do: read a .env file at runtime. Anything uploaded to S3 is public to anyone who can fetch the bucket. Backend-only secrets must be loaded on EC2, not baked into static JS (except public Cognito identifiers).

PKCE + state validation

  • State compared on callback to block CSRF
  • No client secret in token exchange

HTTPS: why S3 alone is not enough

Cognito requires HTTPS callback URLs in production. S3 website endpoints are HTTP only. CloudFront gives you the HTTPS domain Cognito expects.

Browser
→
CloudFront (HTTPS)
CloudFront origin
→
S3 bucket (static files)

Create CloudFront distribution

Origin = your S3 bucket. Default root object = index.html. Note the CloudFront domain, e.g. d1234abcdef.cloudfront.net.

Set Cognito callback URLs

In Cognito → App client → Hosted UI settings:

Callback URL:  https://YOUR_CLOUDFRONT_DOMAIN/cognito-callback.html
Sign-out URL:  https://YOUR_CLOUDFRONT_DOMAIN/newHome.html

Keep http://localhost:3001/cognito-callback.html for local testing if needed.

Fix API calls — tokens must not travel over HTTP

Your HTML pages call http://54.196.133.103:8081. Bearer tokens over plain HTTP can be intercepted. For the assignment, at minimum explain this gap. For a stronger demo, put HTTPS in front of the API (nginx + cert, or ALB).

Common failure: Cognito login succeeds locally but fails from S3 because the callback URL is still HTTP or points to the wrong domain. Match the exact URL Cognito has registered — path included.

Secrets Manager on EC2

The assignment asks for Secrets Manager. Your code currently reads secrets from .env on EC2 — that is better than hardcoding in source, but not the evidence the rubric wants. Wire Secrets Manager conceptually (or in code) and screenshot the secret name, not the value.

What to store

  • sp-games/db-password → DB_PASSWORD
  • sp-games/jwt-secret → JWT_SECRET

Cognito IDs can stay in .env — they are not secret.

EC2 IAM role

Attach a role to the EC2 instance with permission:

secretsmanager:GetSecretValue

No AWS access keys in code. The instance role is the trust boundary.

Startup pattern (concept)

// Backend loads once at startup — frontend never calls this.
const dbPassword = await getSecret("sp-games/db-password");
const jwtSecret  = await getSecret("sp-games/jwt-secret");

process.env.DB_PASSWORD = dbPassword;
process.env.JWT_SECRET  = jwtSecret;
// then start Express as normal
Screenshot tip: AWS Console → Secrets Manager → your secret → show the secret name and ARN with the value column hidden or never revealed. That proves configuration without leaking credentials.
