How to do the AWS part without getting lost

Your assignment asks for AWS S3 frontend hosting, backend hosting, free hosting, Secrets Manager, and Cognito. This lesson teaches the practical route: what each service does, what to configure, and what screenshots prove it.

The Mental Model

AWS is not one magic upload button. Split the app into static files, running server code, secrets, and authentication.

Key idea

S3 hosts the frontend only

S3 can serve files like index.html, login.html, JavaScript, CSS, and images. It does not run Express routes or connect directly to MySQL.

Key idea

The backend must run elsewhere

Your Node.js API needs a runtime host. That host reads secrets from environment variables or Secrets Manager and talks to the database.

The Assignment Route

If you want the most realistic student path, do these in order. Stop after any completed part and still write it clearly in the report.

Prepare the frontend

Find the static frontend folder. For this repo, start from Assignment/FrontEndServer/public. The files inside are what S3 can host.

Deploy static files to S3

Create an S3 bucket, enable static website hosting, set the index document to index.html, upload the frontend files, and open the website endpoint.

Host the backend API

Use a service that can run Node.js. For AWS-only evidence, use EC2 free tier or Elastic Beanstalk if allowed. For assignment practicality, confirm with your lecturer whether a free non-AWS backend host is acceptable.

Move secrets out of code

Database password and JWT secret should not be hard-coded. Store them in Secrets Manager or, at minimum, environment variables for a local/free demo.

Explain Cognito

Cognito replaces custom user signup/login in a production version. You can create a user pool and show the hosted login page as evidence, even if you do not fully rewrite the app.

Part 1: S3 Frontend Hosting

This is the cleanest AWS feature to complete because your frontend is mostly static files.

Validation: For this assignment, S3 is the best frontend hosting answer because the brief explicitly says “Hosting frontend in AWS S3.” For a production static site, AWS also recommends Amplify Hosting because it gives a managed CDN and HTTPS URL on top of the static hosting workflow.

Console steps

  1. Open AWS Console → S3 → Create bucket.
  2. Choose a globally unique bucket name, for example sp-games-yourname-demo.
  3. Keep the region consistent with your other services.
  4. Upload the contents of Assignment/FrontEndServer/public.
  5. Go to Properties → Static website hosting → Enable.
  6. Set index document to index.html.
  7. If using public website hosting, configure bucket policy carefully so only public frontend files are readable.

Evidence to capture

Report sentence: “The frontend was deployed as static files to Amazon S3. S3 is suitable for this layer because it serves HTML, CSS, JavaScript, and assets, while backend API execution remains on a Node-capable host.”
From Learner Lab: After AWS Console opens, search S3 in the top search bar. If S3 is blocked by lab permissions, capture that error and ask your lecturer whether Learner Lab restrictions count as a valid limitation. Most AWS Academy labs allow common services, but each course can restrict services differently.

Part 2: Backend Hosting

The backend is different from the frontend. It needs a live Node.js process, environment configuration, and database access.

AWS route

EC2 free tier

Launch a small Ubuntu EC2 instance, install Node.js, clone or upload the backend, set environment variables, run npm install, then start the server with a process manager such as pm2.

cd Assignment/BackEndServer
npm install
DB_HOST=... DB_USER=... DB_PASSWORD=... JWT_SECRET=... npm start
Simpler route

Document the architecture

If time is tight, show S3 live and explain that the backend requires a Node runtime, not S3. Then include screenshots of local API testing and a deployment plan.

Do not put database credentials in frontend JavaScript. Anything in S3 can be viewed by users. Secrets belong on the backend side only.

Part 3: Secrets Manager

Secrets Manager is the assignment’s “store the secret safely” item. The important learning is that secrets leave source code and are fetched by trusted backend code.

What to store

  • DB_PASSWORD
  • JWT_SECRET
  • Any API keys used by backend code

What not to screenshot

  • The unmasked secret value
  • AWS access keys
  • Database passwords

Backend pattern

The backend should load secrets at startup, then use them like environment config. This is the concept your report should explain.

// Concept only: backend loads secret, then connects safely.
const dbPassword = await loadSecret("sp-games/db-password");
const jwtSecret = await loadSecret("sp-games/jwt-secret");

// Frontend never sees these values.

Part 4: Cognito

Cognito is cloud authentication. In a production redesign, it would handle signup, login, password reset, and token issuing instead of storing plaintext or weakly handled passwords in your own users table.

Hosting vs auth: Cognito is not a hosting service. Your teammate is right that Cognito is important, but it answers the authentication requirement, while S3 or Amplify answers frontend hosting and EC2/Elastic Beanstalk/another Node host answers backend hosting.
Current app concept Cognito version Security improvement
App stores users and passwords itself Cognito User Pool stores identities Managed password flows and token issuing
Custom login endpoint returns JWT Cognito Hosted UI signs users in Less custom authentication code to get wrong
Backend trusts app-issued token Backend verifies Cognito-issued token Clear issuer, expiry, and claims model
Realistic scope: For this assignment, you can show a Cognito user pool and hosted login page, then explain how the backend would verify Cognito JWTs. A full migration is useful but may be bigger than the optional marks require.
From Learner Lab: Search Cognito in the AWS Console, create a User pool, choose email sign-in, create an app client, and open the hosted UI or managed login page if the lab allows it. Screenshot the user pool overview and login page, not any sensitive client secrets.

Full app migration path

Frontend Cognito code belongs in Assignment/FrontEndServer/Public. Backend token verification belongs in Assignment/BackEndServer/auth/verifyToken.js. Do not put Cognito secrets or database secrets in S3.

HTTPS path for S3

S3 website endpoints are HTTP only. Put CloudFront in front of S3, use the CloudFront HTTPS domain as the Cognito callback URL, and upload the updated frontend files to S3.
