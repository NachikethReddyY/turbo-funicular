05OWASP Vulnerability Map for ST2515 Report

Each vulnerability found in the codebase mapped to an OWASP Top 10 category — ready for the assignment report. The assignment brief requires 5 categories × 2 vulnerabilities (1 detailed + 1 brief).

A01 — Broken Access Control

1. Missing auth on POST /category and POST /platform

These endpoints lack the verifyToken middleware. Any unauthenticated user can call them. The frontend checks admin role client-side, but the backend never re-verifies.

controller/app.js:87-93, model/category.js, model/platform.js

2. Missing auth on DELETE /game/:id

Game deletion has no auth middleware — anyone can delete games via a direct API call.

controller/app.js:110
A02 — Cryptographic Failures

1. Passwords stored in plaintext

No hashing (bcrypt/argon2) is used when storing or verifying passwords. If the DB is breached, all credentials are exposed.

model/users.js:30-40

2. JWT secret hardcoded in config.js

The signing key 'Assignment2key' is stored in plaintext in source control. Anyone with repo access can forge tokens.

config.js
A03 — Injection (SQLi)

1. String-interpolated SQL in game.js

User input is concatenated directly into SQL strings instead of using parameterised queries. Example: `SELECT * FROM game WHERE title LIKE '%${searchTerm}%'`

model/game.js:20-80

2. Interpolated queries in users.js, category.js, platform.js

Multiple model files build SQL via template literals with unsanitised user input — classic SQL injection.

model/users.js, model/category.js, model/platform.js
A04 — Insecure Design

1. Client-side admin gating

Admin functionality is hidden client-side via JS (hiding .admin-tab elements). The backend doesn't verify admin role for several privileged operations — the design trusts the client.

gamesSearch.html, admin.html, controller/app.js

2. No input validation architecture

No consistent input validation/sanitisation layer exists. Each endpoint handles (or doesn't handle) validation ad-hoc, if at all.

All controllers and models
A05 — Security Misconfiguration

1. DB credentials in source code

MySQL username root and password root are hardcoded in databaseConfig.js.

model/databaseConfig.js

2. CORS fully open

cors() is used without any origin restriction — any website can make API calls to the backend.

controller/app.js:28
A07 — Identification & Authentication Failures

1. No password complexity rules

Registration accepts any password with no length, character, or strength requirements. No MFA available.

register.html, model/users.js

2. rememberMe in localStorage

The "remember me" feature stores credentials/tokens in localStorage — vulnerable to XSS theft.

login.html:95-110
A09 — Security Logging & Monitoring Failures

1. No audit logging

No login attempts, failed auth, or admin actions are logged. Intruders can probe undetected.

Entire backend

2. No request logging middleware

No morgan/winston/pino — no record of incoming requests, IPs, or timestamps exists.

server.js, controller/app.js
